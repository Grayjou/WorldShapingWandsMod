## Project Overview for New Agents / Engineers

WorldShapingWandsMod is a tModLoader mod for Terraria that provides 10 wand families
(Building, Dismantling, Replacement, Wiring, Safekeeping, Coating, Fluids, Torches,
Delimitation, Molding) × 4 selection modes (Instant, Select, Confirm, Stamp) = 40 items,
plus 2 standalone TorchWheel wands, enabling players to edit the world in
geometric shapes (Rectangle, Ellipse, Diamond, Triangle, Elbow, Cardinal Line, Straight
Line — plus half-shapes via SliceMode) with configurable Filled/Hollow modes.

### Key Design Patterns
- Provider pattern: shapes implement IShapeProvider; ShapeRegistry maps enum → provider
- Immutable state: SelectionState and ShapeContext are immutable (factory methods)
- Per-wand settings: each family has its own struct on WandPlayer — no global state
- Client-side safety: all rendering is [Autoload(Side = ModSide.Client)]
- Server-authoritative MP: WandPacketHandler handles client → server → broadcast

### Priority Scheme — `Num1*Num2` two-axis pairs

All session-prompt task priorities use the **explicit two-axis pair** form:

```
P <Assess>*<Impl>: <task text>
```

| Axis        | Question it answers                                                | Lower value = more urgent |
|-------------|--------------------------------------------------------------------|---------------------------|
| `<Assess>`  | "How urgently does the agent need to *think about / read / design* this?" | Investigate first |
| `<Impl>`    | "How urgently does the resulting artifact / code need to *ship this session*?" | Ship first |

The two axes are **independent**. A research-heavy item that needs immediate
consideration but whose implementation depends on user feedback is `P 1*4`
(assess now, ship later). A small tidy-up that only earns its keep by being
one-shot done is `P 3*1` (low subject importance, but if you start it, finish it).

**Decoding cheatsheet:**

| Tag        | Meaning                                                                  |
|------------|--------------------------------------------------------------------------|
| `P 0*0:`   | Drop everything else. Reserved for true emergencies (data loss, build broken). |
| `P 1*1:`   | Standard hot item. Investigate now and ship now.                         |
| `P 1*2:`   | Investigate now; shipping can slip slightly if a higher item bloats.     |
| `P 1*3:`   | Design/research now; implementation likely next session.                 |
| `P 1*4:`   | Assess now; flag explicitly for a future session as a written deferral.  |
| `P 2*1:`   | Ordinary urgency to assess; if you start, finish — don't half-ship.      |
| `P 2*2:`   | Default. Assess and implement at normal pace. Equivalent to bare `P 2:`. |
| `P 2*3:`   | Mild item; written reasoning OK, code can wait.                          |
| `P 3*1:`   | Low-importance topic; one-shot tidy.                                     |
| `P 4*4:`   | Background-noise item — only touch if zero higher items remain.          |

**Shorthand:** `P 2:` expands to `P 2*2:` (single number = both axes equal).
**Uncertainty:** `?` replaces a digit on the axis you are unsure about, e.g.
`P 1*?:` = "assess urgently; ship-cost is unknown — agent must recommend a
final value at session end so the next session can pin it down."

**Agent ordering rule (operationally important):**

1. **Sort goals by `<Assess>` ascending.** All `P 1*?` goals get touched (at
   least investigated) before any `P 2*?` goal.
2. **Within the same `<Assess>`, sort by `<Impl>` ascending.** Among `P 1*?`
   goals, `P 1*1` ships before `P 1*3`.
3. **Every goal must produce an artifact, even if it's a deferral.** A `P 1*4`
   goal produces an investigation note + an explicit deferral entry in
   `dev_notes/planning/DeferredForNextSession.md`.
4. **`?` triggers a recommendation.** A `P 1*?` goal must end the session with
   the agent stating "I recommend this is `P 1*N`" so the user can confirm or
   correct for next time.

**Migration of legacy forms:** Old prompts using bare priority numbers (e.g.
`Priority 0`, `Priority 0.1`, `Priority 1`) still parse. Treat:
- `Priority N` or `Priority: N` → `P N*N:` (both axes equal).
- Sub-priority decimals (`Priority 0.1`, `Priority 0.2`, ...) → relative tie-
  breaker within the same `<Assess>` tier; the digit after the dot maps to
  `<Impl>` (so `Priority 0.1` ≈ `P 0*1`, `Priority 2.3` ≈ `P 2*3`).
- Trailing `*` (the retired conditional-suffix scheme) → `?` on the `<Impl>`
  axis only, e.g. `Priority 2*` → `P 2*2?:`.

This means old session drafts continue to work; only new prompts should adopt
the explicit `P 1*1:` form going forward.

### Naming Conventions (Lexicographic Ordering)

All date-based filenames use **YYYY-MM-DD** format for lexicographic sort:

| File Type | Format | Example |
|-----------|--------|---------|
| **Sessions** | `Sessions/YYYY-MM-DD Session N.txt` | `2026-04-15 Session 1.txt` |
| **Letters (drafts)** | `LetterDrafts/WSW YYYY-MM-DD Letter #N.md` | `WSW 2026-04-23 Letter #5.md` |
| **Letters (sent archive)** | `LettersSent/WSW YYYY-MM-DD Letter #N.md` | `WSW 2026-04-23 Letter #5.md` |
| **Cavendish responses (inbox)** | `dev_notes/inbox/{addressed,unaddressed}/Response_<...>.md` | `dev_notes/inbox/unaddressed/Response_WSW 2026-04-23 Letter #5.md` |
| **Implementation Logs** | `dev_notes/planning/Implemented_YYYY-MM-DD.md` | `Implemented_2026-04-15.md` |

Implementation logs use the **date of the first session** in that log file. Multiple sessions on the same day are sections within the same file (S1, S2, S3...). New days get new files.

Legacy files (ImplementedSession9.md through ImplementedSession16.md) are kept as-is — only new logs use the date format.

**Format buckets** (all five sections present even when empty; no silent
omissions — empty buckets carry an explicit *"None this session."* line):

```markdown
## 🟥 Blocking — needed before specific downstream work can begin
## 🟧 Greenlight asks — work is designed and waiting on a yes/no
## 🟨 Clarifications — small ambiguities that block clean execution
## 🟩 Long-running ratifications — process items waiting for one nod
## ✅ Closed this session — items resolved during this session
```

Each item is concrete, actionable, and bounded — *"ratify X"* or *"supply Y"*
or *"decide between A/B"* — never an open-ended *"think about Z."*

**Discipline**:

- Fill in **during** the session as items surface; do not batch at end.
- GrayJou reads the file BEFORE writing the next session prompt. Treat as the
  primary single-pane-of-glass view of his outstanding queue.
- **Roll-forward at session start**: before doing any new work, open the file,
  scan every bucket, drop items GrayJou cleared (in his most-recent prompt or
  letter), age forward items he didn't address (tag `[rolled S<prev> → S<this>]`
  inline so staleness is visible at a glance), add new items as session work
  surfaces them.
- Closed items move to the ✅ bucket DURING the session that closes them; at
  the next session's housekeeping pass, the ✅ bucket is cleared (those items
  are "sealed" — anyone needing archaeology pulls the closed item from that
  session's `Implemented_*.md` log or letter).
- No silent skips — if a session has nothing pending, all five buckets carry
  *"None this session."* and the file ships unchanged otherwise.
