## 1. Project Overview

**WorldShapingWandsMod** is a [tModLoader](https://github.com/tModLoader/tModLoader) mod for
Terraria that provides geometric world-shaping tools. Players can build, dismantle, replace,
wire, and protect large areas of the world using customisable shapes — from rectangles and
ellipses to triangles, elbows, and half-ellipses — with a single drag or series of clicks.

- **Platform**: tModLoader for Terraria (targets .NET 8.0)
- **Status**: Pre-release / Active Development (not yet on Steam Workshop)
- **Version**: 0.1.0 (`build.txt`)
- **Root mod class**: `WorldShapingWandsMod.cs` — singleton `Instance`, initialises `ShapeRegistry`

### Key Design Principles

1. **Immutable state** — `SelectionState` and `ShapeContext` are immutable. Every mutation returns
   a new instance, preventing aliasing bugs.
2. **Provider pattern** — Shapes implement `IShapeProvider`; `ShapeRegistry` maps enum values to
   providers. Adding a new shape = one class + one registry entry.
3. **Separation of concerns** — Shapes produce filled tiles only; mode transformation
   (Filled/Hollow) is handled by `OutlineHelper`. Settings, selection, execution, and rendering
   are separate subsystems.
4. **Per-wand settings** — Each wand family has its own settings struct, stored on `WandPlayer`.
   No global state leaks between families.
5. **Client-side safety** — All rendering systems (`SelectionOverlay`, `SafekeepingOverlay`,
   `WandUISystem`) are `[Autoload(Side = ModSide.Client)]`.

## 4. Shape System Deep-Dive

### 4.1 Provider Registry

`ShapeRegistry` maintains a `Dictionary<ShapeType, IShapeProvider>`. Initialised in
`WorldShapingWandsMod.Load()`, cleared in `Unload()`.

```
Registered providers:
  ShapeType.Rectangle    → RectangleShape
  ShapeType.Ellipse      → EllipseShape
  ShapeType.Diamond      → DiamondShape
  ShapeType.Triangle     → TriangleShape
  ShapeType.Elbow        → ElbowShape
  ShapeType.CardinalLine → CardinalLineShape
  ShapeType.HalfEllipseH → HalfEllipseHShape
  ShapeType.HalfEllipseV → HalfEllipseVShape
```

### 4.2 Shape Algorithms

All shapes implement `IShapeProvider.GetTiles(ShapeContext)` returning a `ShapeTileSet`.
Shapes only produce **filled** tiles; `OutlineHelper` converts to Hollow mode afterward.

#### Rectangle
Trivial: iterates every `(x, y)` within `[Start, End]` bounds.

#### Ellipse
Direct `Math.Sqrt` rasterisation: for each `(x, y)` in bounds, checks
`((x - cx)² / rx²) + ((y - cy)² / ry²) ≤ 1.0` where `(cx, cy)` is the centre and
`(rx, ry)` are the semi-axes.
Dev: I researched algorithms of EllipseRasterization, it turns out that the most efficient algorithm is only 20% faster than direct sqrt calculation.  Check my [Ellipse Rasterization Benchmarks](https://github.com/Grayjou/EllipseRasterization)

#### Diamond
Integer arithmetic with ×2 scaling to avoid floating-point: uses Manhattan distance from centre.
A point is inside if `|2*(x - cx)| * H + |2*(y - cy)| * W ≤ W * H` (scaled to avoid division).

#### Triangle
Computes a filled triangle anchored at a bias-determined vertex. Uses horizontal or vertical
scanline fill depending on orientation. Handles degenerate cases (1×1, 1×N, N×1) explicitly
to avoid division-by-zero.

#### Elbow (formerly "Edge")
L-shaped right-angle joint: one horizontal segment and one vertical segment meeting at a corner.
The `VerticalFirst` flag determines which segment is drawn first — critical for the wiring
overlay axis-flip fix (#39).

#### CardinalLine
8-sector direction detection for straight lines snapped to the 8 cardinal/ordinal directions.
Walks from start to end along the dominant axis using Bresenham-like stepping.

#### HalfEllipseH / HalfEllipseV
Generates a full ellipse at doubled dimensions, then clips to one half (horizontal or vertical).
The clipping half is determined by the `VerticalFirst` or bias settings.

### 4.3 OutlineHelper

Central mode transformation system. Shapes call `OutlineHelper` after producing filled tiles.

- **Filled mode** (`ShapeMode.Filled`): returns the tile set as-is.
- **Hollow mode** (`ShapeMode.Hollow`): applies Chebyshev erosion — removes interior tiles that
  are surrounded on all 8 sides by other tiles (to a configurable thickness depth).
  - Thickness 0: slim 1-pixel boundary
  - Thickness 1: standard boundary
  - Thickness 2+: repeated erosion passes, with early exit if the tile set becomes empty

### 4.4 ShapeContext

Immutable struct encapsulating all parameters for a single shape computation:

| Field | Type | Description |
|---|---|---|
| `Start` | `Point` | Top-left corner (tile coordinates) |
| `End` | `Point` | Bottom-right corner (tile coordinates) |
| `Mode` | `ShapeMode` | Filled or Hollow |
| `Thickness` | `int` | Hollow boundary thickness (0–50) |
| `HBias` | `HorizontalBias` | None / Left / Center / Right |
| `VBias` | `VerticalBias` | None / Top / Center / Bottom |
| `VerticalFirst` | `bool` | For Elbow/Cardinal: which axis draws first |
| `EqualDimensions` | `bool` | Force square bounding box (circles, squares, equilateral diamonds) |

`GetBounds()` normalises Start/End and, when `EqualDimensions` is true, expands the smaller
dimension to match the larger one (anchored at Start).

## 5. Selection State Machine

### States

`SelectionState` is an immutable sealed class with factory methods:

| Factory | Meaning |
|---|---|
| `SelectionState.Empty()` | No active selection |
| `SelectionState.Create(start)` | Selection started at `start` |
| `SelectionState.WithEnd(end, wasClamped, verticalFirst)` | End point set/updated |
| `SelectionState.WithLocked()` | Selection locked (ThreeClick confirm phase) |

### Lifecycle per SelectionMode

**OneClick (Instant)**:
1. Mouse down → `Create(cursor)` → `IsActive = true`
2. Mouse move → `WithEnd(cursor)` (drag updates end)
3. Mouse up → Execute → `Empty()`

**TwoClick (Select)**:
1. Click 1 → `Create(cursor)`
2. Mouse move → `WithEnd(cursor)` (live preview)
3. Click 2 → Execute → `Empty()`

**ThreeClick (Confirm)**:
1. Click 1 → `Create(cursor)`
2. Mouse move → `WithEnd(cursor)` (live preview)
3. Click 2 → `WithLocked()` (end point frozen)
4. Click 3 → Execute → `Empty()`

**FourClick (Stamp)**:
1. Click 1 → `Create(cursor)`
2. Click 2 → `WithLocked()` + record `StampDelta`
3. Click 3+ → Stamp at new position (Execute with offset), state remains locked
4. Right-click → Cancel → `Empty()`

### Cross-Wand Guard

`WandPlayer.SelectionOwnerItemType` tracks which wand item started the selection. If the player
switches to a different wand mid-selection, `PostUpdate()` detects the mismatch and auto-cancels
with a chat message.

### Dimension Capping

`WandPlayer.ClampEndToCaps()` enforces per-shape selection limits from `WandConfig`:
- **SmallCap** (default 1000): for Elbow and CardinalLine
- **BigCap** (default 200): for all other shapes

When clamped, `SelectionState.WasClamped` is set to `true`, triggering an orange flash in the
overlay.

## 7. Wand Execution Pipelines

### Building (`WandOfBuildingBase`, ~514 lines)

1. Compute `ShapeContext` from selection + settings
2. `ShapeRegistry.GetShapeTiles(context)` → tile set
3. Count available items via `ItemTypeHelper.CountItems(player, placeType)`
4. Check infinite-resource mode (`WandConfig.EnableInfiniteResource`)
5. For each tile in the set:
   - Skip if `SafekeepingSystem.IsTileProtected(x, y)`
   - Skip if position already occupied (unless overwrite mode)
   - Find matching item in inventory (`ItemTypeHelper.FindFirstItem`)
   - Snapshot tile state for undo
   - `WorldGen.PlaceTile(x, y, tileType)` or `WorldGen.PlaceWall(x, y, wallType)`
   - Apply slope if configured
   - Consume item from inventory (unless infinite resource)
   - Handle exhaustion mode if stock runs out mid-operation
6. `BulkTileOperations.BatchFrameUpdate` + `BatchNetworkSync`
7. `UndoManager.CommitAction(snapshots)`
8. Chat message with placed/skipped counts

**Wall building path**: When `PlaceType.Wall` is selected, uses `WorldGen.PlaceWall` instead of
`PlaceTile`. The item search uses `createWall > 0` condition.

### Dismantling (`WandOfDismantling`, ~350 lines)

1. Compute `ShapeContext` from selection
2. Get tile set from registry
3. Begin undo action
4. For each tile in the set:
   - Skip if protected
   - If `DestroyTiles` and tile exists: validate pick power + `CanKillTile`, snapshot, `WorldGen.KillTile`
   - If `DestroyWalls` and wall exists: snapshot (if not already), `WorldGen.KillWall`
   - Single snapshot per tile (HashSet guard prevents double-snapshot corruption)
5. Batch frame update + network sync
6. Commit undo
7. Optional: enqueue for progressive processing if enabled

### Replacement (`WandOfReplacementBase`, ~550 lines)

1. Compute shape context
2. Get tile set
3. If OldObject or NewObject is Wall → route to `ExecuteWallReplacement()`
4. For each tile (tile mode):
   - Skip if protected
   - Check if tile matches `OldObject` type (or empty if OldObject = Air)
   - Validate pick power + `CanKillTile` on existing tile
   - Snapshot for undo
   - `WorldGen.ReplaceTile` (preferred) with `KillTile` + `PlaceTile` fallback
   - Consume replacement item from inventory
5. Batch operations + undo commit
6. Sound effect: `SoundID.Item29` on success

**Wall replacement path** (`ExecuteWallReplacement`):
1. Resolve source/target wall items from inventory using `createWall`
2. For each tile in the set:
   - Skip if wall-protected
   - Match `tile.WallType` against source wall type (variant-aware via `ResolveReplacementPair`)
   - If target is Air: `WorldGen.KillWall` (erase wall)
   - Otherwise: `WorldGen.KillWall` + `WorldGen.PlaceWall` + `Framing.WallFrame`
   - Consume wall item from inventory
3. Batch operations + undo commit

**Substrate variant detection**: `ItemTypeHelper` maps base tiles to their variants. Replacing
"dirt" catches all grass-covered variants (dirt, grass, jungle grass, mushroom grass, etc.).

### Wiring (`WandOfWiringBase`, ~190 lines)

1. Compute shape context (default shape: Elbow)
2. Get tile set
3. For each tile:
   - Skip if protected
   - For each enabled wire colour: `WiringHelper.PlaceWire` or `RemoveWire`
   - For actuators: `WiringHelper.PlaceActuator` or `RemoveActuator`
4. Network sync

**Note**: Wire/actuator inventory consumption is implemented (#113). Uses per-wire and
per-actuator infinite resource checks via `WiringHelper`. Full multiplayer packets via
`WandPacketHandler` (#112).

### Safekeeping (`WandOfSafekeepingBase`, ~230 lines)

1. Compute shape context
2. Get tile set
3. For each tile:
   - If Protect mode: `SafekeepingSystem.ProtectTile/ProtectWall`
   - If Unprotect mode: `SafekeepingSystem.UnprotectTile/UnprotectWall`
4. Chat message with count

**No undo** for safekeeping operations — protection is toggled, not destructive.

### Coating (`WandOfCoatingBase`)

1. Compute `ShapeContext` from selection + settings
2. Get tile set from registry
3. For each tile:
   - Skip if protected
   - If Apply mode: apply the selected coating (`TileCoating.Illuminant`, `Echo`, or `Actuated`)
   - If Remove mode: remove the selected coating from the tile
4. `SendTileSquare` bounding-box sync for multiplayer
5. Chat message with applied/skipped counts

**Multiplayer**: Currently uses `SendTileSquare` bounding-box sync. Proper packet-based
sync is planned as part of the 7-day MP fix schedule (see `MultiplayerFixSchedule.md`).

## 9. Overlay & Rendering Systems

### SelectionOverlay (`Common/Drawing/SelectionOverlay.cs`, ~400 lines)

`ModSystem` registered `[Autoload(Side = ModSide.Client)]`. Renders in `PostDrawTiles`.

**Features**:
- **Cached tile set computation** — `_cacheVerticalFirst`, `_cacheShapeInfo`, `_cacheStart`,
  `_cacheEnd` fields prevent recomputation when nothing changed. Critical for performance
  since `GetTiles` can be expensive for large shapes.
- **Screen culling** — only tiles within the viewport (plus margin) are rendered.
- **3 render modes** — configurable via `OverlayRenderMode` in `WandConfig`:
  - `AlwaysFullShape`: semi-transparent fill + brighter outline (default)
  - `OnlyOutline`: outline border only, no fill
  - `OutlineAndPartialFill`: outline + sparse interior fill
- **Dimension label** — `W × H  (count)` text rendered near the cursor (UI-scaled, screen-clamped
  with Math.Clamp to all 4 edges, offset 24×−32 from cursor).
- **Debounced area count** — tile count appended after 10 frames of stable dimensions (zero drag
  cost). Cache fields: `_areaStableFrames`, `_lastAreaDimensions`, `_cachedAreaCount`.
- **Shape-specific dimensions** — `ElbowShape` overrides `GetDisplayDimensions` to show segment
  lengths instead of bounding box.
- **Clamped indicator** — orange flash when `WasClamped` is true.
- **Cancelled selection overlay** — when a selection is cancelled, the outline fades from the
  wand's cancel colour over ~45 ticks with rising "Cancelled" text.
- **Flicker-free AlwaysFullShape** — fade-in animation no longer triggers on every cursor movement (#146).

**Wand colours** are defined in `WandColors.cs` (static palette per wand family).

### SafekeepingOverlay (`Common/Drawing/SafekeepingOverlay.cs`)

`ModSystem` registered `[Autoload(Side = ModSide.Client)]`. Renders in `PostDrawTiles`.

**Behaviour**: when the player holds any Wand of Safekeeping, draws coloured squares over
protected positions:

| Visual | Meaning |
|---|---|
| **Cyan** | Tile-only protection |
| **Magenta** | Wall-only protection |
| **Gold** | Both tile and wall protected |

- Low opacity (fill: 22%, outline: 45%) to avoid obscuring the world.
- Uses `TextureAssets.MagicPixel` — no custom assets needed.
- Screen-culled for performance.
- No separate toggle — activates automatically when holding the wand.

### CancelledSelectionState (`Common/Selection/CancelledSelectionState.cs`)

Snapshot class for the cancel animation:
- Stores `Start`, `End`, `VerticalFirst`, `Shape`, `CancelColor`, `Duration`
- `Opacity` decays over time; `IsExpired` triggers removal
- Each wand family has its own cancel colour (defined in `WandColors.cs`)

## 10. Undo System

### UndoManager (`Common/Undo/UndoManager.cs`)

`ModPlayer` with a `Stack<UndoAction>` capped at 20 entries.

| Method | Description |
|---|---|
| `BeginAction()` | Creates a new `UndoAction` |
| `CommitAction(snapshots)` | Pushes the action with tile snapshots onto the stack |
| `Undo()` | Pops and restores the top action |
| `ClearHistory()` | Empties the stack (called on `OnEnterWorld`) |

When the stack exceeds 20, the **oldest** action is dropped (FIFO eviction — this was a bug
fix; originally dropped the newest).

### UndoAction & TileSnapshot (`Common/Undo/UndoAction.cs`)

`TileSnapshot` captures per-tile state before modification:
- Position (`Point`)
- `HasTile`, `TileType`, `TileFrameX/Y`, `Slope`, `IsHalfBlock`
- `WallType`

`Restore()` writes the snapshot back to `Main.tile[x, y]` and calls `NetMessage.SendTileSquare`
for multiplayer sync.

### Single-Snapshot Guard

The dismantling wand's execution loop uses a `HashSet<Point>` to ensure each tile gets exactly
one snapshot, even when both tile and wall destruction apply to the same position. This prevents
the "double snapshot" bug where the second snapshot captures the already-destroyed tile state.

## 11. Safekeeping (Protection) System

### SafekeepingSystem (`Common/Systems/SafekeepingSystem.cs`, 117 lines)

`ModSystem` that stores two `HashSet<Point>` collections:
- `_protectedTiles` — tile positions that cannot be modified
- `_protectedWalls` — wall positions that cannot be modified

**API**:
- `IsTileProtected(x, y)` / `IsWallProtected(x, y)` — O(1) lookup
- `ProtectTile(x, y)` / `ProtectWall(x, y)` — add to set
- `UnprotectTile(x, y)` / `UnprotectWall(x, y)` — remove from set
- `ClearAll()` — empties both sets

**Persistence**: saved and loaded via `TagCompound` using flat `int[]` arrays
(`[x1, y1, x2, y2, ...]`). Data survives server restarts and world reloads.

**Integration**: all `Execute*` methods in the 5 non-safekeeping wands check
`SafekeepingSystem.IsTileProtected(x, y)` before modifying each tile, skipping protected
positions and incrementing a `skippedProtected` counter reported in the chat message.

## 12. Progressive Processing System

### ProgressiveTileProcessor (`Common/Systems/ProgressiveTileProcessor.cs`, 368 lines)

`ModSystem` that enables timed batch processing for large operations, activated via
`WandConfig.EnableProgressiveMode`.

**Design**:
- `EnqueueDismantling(tiles, player)` / `EnqueueReplacement(tiles, player, settings)`
- `PostUpdateWorld` processes batches on a configurable interval
- Default: 400 tiles per batch, every 0.3 seconds (18 ticks at 60 TPS)
- Supports undo — snapshots are taken per batch
- Visual feedback: progress messages in chat

**Use case**: prevents frame drops for very large operations (thousands of tiles). Without
progressive mode, a 10,000-tile dismantling operation would freeze the game for multiple frames.

## 15. Dual Sprite System

### Problem
Terraria uses a single texture for both world rendering and inventory display. Wands need
different sprites for each context (detailed world sprite vs clear inventory icon).

### Solution

`BaseCyclingWand` maintains a static `Dictionary<int, Asset<Texture2D>>` mapping item `Type`
to inventory textures. On `SetDefaults`, if an `_Inventory.png` asset exists for the item,
it is loaded and registered.

- **`PreDrawInInventory`** — swaps `TextureAssets.Item[Type]` to the inventory texture before
  drawing, then restores it.
- **`PreDrawInWorld`** — ensures the world texture is active (restores if the inventory texture
  was left active).
- **Zero cost** if `_Inventory.png` doesn't exist — the dictionary entry is simply not created.

## 21. Known Limitations & Stale Files

### Active Limitations

1. **No multiplayer packets for 5 families** — Only Wiring has proper packet-based MP sync.
   Building, Dismantling, Replacement, Safekeeping, and Coating rely on `WorldGen.gen` hacks
   or `SendTileSquare` bounding-box sync. See `MultiplayerFixSchedule.md` for the 7-day fix plan.
2. **Undo doesn't handle multi-tile objects well** — Doors, furniture, and other multi-tile
   objects may enter broken states when partially restored.
3. **`ObjWall.png` icon missing** — Building wand's Wall button uses `ObjSolid.png` as placeholder.
4. **`WandSettings.cs` is partially legacy** — Global settings struct still used by the overlay
   system but most per-wand logic uses the family-specific settings.
5. **Safekeeping has zero MP code** — Protection state is purely server-local. Client/server
   discrepancy in MP means one side may have stale protection data.

## 23. Conventions & Patterns

### Naming Conventions
- **Classes**: PascalCase (`BaseCyclingWand`, `ShapeRegistry`)
- **Enums**: PascalCase values (`ShapeType.Elbow`, `SelectionMode.FourClick`)
- **Private fields**: `_camelCase` prefix (`_protectedTiles`, `_cacheVerticalFirst`)
- **Constants**: PascalCase (`CommonLore`, `MaxUndoActions`)
- **File names**: match class name exactly
- **Historical renames**: `WandOfDestruction` → `WandOfDismantling` (#15), `Edge` → `Elbow` (#38),
  `Protection` → `Safekeeping` (#40), `ThicknessControls` → `WandControls`

### Architecture Patterns
- **Provider pattern**: `IShapeProvider` → `ShapeRegistry` — shapes are pluggable
- **Immutable state**: `SelectionState`, `ShapeContext` — factory methods return new instances
- **Per-wand settings**: family-specific structs on `WandPlayer` — no global state
- **Static utility classes**: `ItemTypeHelper`, `GeometryHelper`, `BulkTileOperations` — pure
  functions, no game-state mutation
- **ModSystem for singletons**: `WandUISystem`, `SafekeepingSystem`, `SelectionOverlay`,
  `ProgressiveTileProcessor` — tModLoader lifecycle-managed
- **ModPlayer for per-player state**: `WandPlayer`, `UndoManager` — isolated per player

### Safety Patterns
- **World-bounds guard**: `WorldGen.InWorld(x, y, 1)` before every tile access
- **Pick-power validation**: `HasEnoughPickPowerToHurtTile` + `CanKillTile` before destruction
- **Support detection**: `WouldTileLoseSupport` prevents replacing tiles that would cause
  structural collapse of walls or torches (#145)
- **Demon altar protection**: `AllowDemonAltarDestruction` config prevents accidental altar removal (#124)
- **Single snapshot per tile**: `HashSet<Point>` guard in dismantling loop
- **Client-side autoload**: all rendering tagged `[Autoload(Side = ModSide.Client)]`
- **Null guards**: UI system null-checks panels in `CloseAllUI` and `ModifyInterfaceLayers`
- **Network sync**: `NetMessage.SendTileSquare` on all tile modifications when `netMode == MultiplayerClient`

### Adding a New Shape

1. Create `Common/Geometry/Shapes/NewShape.cs` implementing `IShapeProvider`
2. Add value to `ShapeType` enum
3. Register in `ShapeRegistry.Initialize()`
4. Add icon asset to `Assets/Icons/` (e.g., `ShapeNewFilled.png`, `ShapeNewHollow.png`)
5. Add icon buttons to all relevant settings panels
6. Add to `WandSettingsState` shape cycling arrays
7. Add localization key in hjson file
8. Add cap check in `WandPlayer.ClampEndToCaps()` if needed

### Adding a New Wand Family

1. Create `WandOf*Settings.cs` in `Common/Settings/`
2. Add settings field to `WandPlayer`
3. Create `WandOf*Base.cs` in `Content/Items/` (abstract, extends `BaseCyclingWand`)
4. Create 4 concrete mode classes (Instant/Select/Confirm/Stamp)
5. Override `WandLore` for lore text
6. Create sprites: 4 main + 4 `_Inventory` PNGs
7. Create `*SettingsPanel.cs` in `Common/UI/`
8. Register panel in `WandUISystem`
